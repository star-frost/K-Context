# 测试与评估规格

## 功能验收

功能验收确认系统是否完成本地知识库助手的核心闭环。

| 验收对象 | 证据形态 |
|---|---|
| PDF 导入 | 支持导入并生成 Document 元数据 |
| Markdown 导入 | 支持导入并生成 Document 元数据 |
| TXT 导入 | 支持导入并生成 Document 元数据 |
| 格式拒绝 | 对未列明格式给出明确错误提示 |
| 解析 | 支持格式均可转换为 DocumentBlock IR |
| 清洗 | 空文本和噪声块按规则处理，结构信息保留 |
| 切块 | 基于 IR 生成可回溯 Chunk |
| 索引 | 已生成 Chunk 可进入本地检索索引 |
| 检索 | 用户问题可召回相关 Chunk |
| 回答 | 回答基于召回 Chunk 生成 |
| 引用 | 回答返回 Citation 列表 |
| 证据不足 | 依据不足时给出明确提示 |
| 桌面端界面 | 会话区、主问答区、引用区可见 |

## 结构验收

结构验收关注系统是否保持可维护边界。

- 文档处理、检索、LLM 调用、会话逻辑、UI 有清晰模块边界。
- 领域对象有明确输入输出，不完全依赖隐式字典传递。
- LLM 调用通过独立服务封装。
- 业务层不直接绑定具体厂商 SDK。
- 从任一 Citation 能追溯到 Chunk、DocumentBlock 和来源文档位置。

## RAG 流程测试

每个知识库问答用例应覆盖以下链路：

1. 接收用户问题。
2. 构造检索查询。
3. 召回 Top-K Chunk。
4. 组装上下文。
5. 调用 LLM 服务。
6. 生成回答。
7. 返回回答与引用。

若第 3 步或第 7 步缺失，该用例不满足 RAG 流程要求。

## 引用与证据不足测试

引用测试应确认：

- 引用包含来源文档名。
- 引用包含 Chunk 标识。
- 引用包含页码或等价位置。
- 引用包含文本片段。
- 引用可回溯到来源文档。

证据不足测试应确认：

- 检索为空时，系统明确说明当前知识库依据不足。
- 召回内容不足以支撑结论时，系统不伪造确定性回答。
- 引用列表为空时，回答正文必须体现依据不足。

## UI 验收

桌面端界面至少应展示：

- 会话区：展示和切换多个聊天会话。
- 主问答区：展示用户问题、系统回答和输入框。
- 引用区：展示引用来源、文档名、位置和引用片段。
- 文档导入入口：允许用户选择本地 PDF、Markdown、TXT。
- 处理状态提示：展示已导入、已解析、已切块、已建立索引、处理失败等状态。

## 评估指标定义

评估指标用于论文中的优化前后对比，不作为通过条件。

| 指标 | 含义 | 记录方式 |
|---|---|---|
| 检索准确率 | 检索结果命中人工标注依据的程度 | 基于测试问题集和期望依据记录 |
| 响应延迟 | 从用户提交问题到回答与引用返回的耗时 | 记录端到端时间 |
| Token 消耗 | 单次回答调用中的输入、输出和合计 Token 使用 | 从 LLM 服务调用元数据记录 |

## 测试集要求

测试集应包含：

- 覆盖 PDF、Markdown、TXT 的样例文档。
- 与文档内容相关的问题。
- 每个问题对应的期望文档或期望片段说明。
- 至少一类证据不足问题。
- 能体现结构边界的材料，例如标题、列表、代码块、跨页内容或纯文本段落。

测试集不得包含与当前知识库无关却要求系统伪造答案的问题。

## 优化前后对比方法

对比方法用于体现系统改进过程：

- 固定测试问题集和样例文档。
- 固定记录口径，包括检索结果、回答引用、延迟和 Token 消耗。
- 在优化前后分别运行同一测试集。
- 对比检索命中情况、端到端耗时、Token 使用变化。
- 记录变化方向和原因说明。

该方法只要求可复现、可比较、可解释，不以某个具体数值作为通过条件。

## 论文表格输出要求

论文表格建议包含：

| 字段 | 含义 |
|---|---|
| 问题编号 | 测试问题标识 |
| 文档类型 | PDF、Markdown 或 TXT |
| 期望依据 | 人工标注的文档或片段 |
| 召回依据 | 系统返回的 Chunk 或文档片段 |
| 依据匹配说明 | 用文字说明召回依据与期望依据的关系 |
| 响应延迟 | 端到端耗时记录 |
| Token 消耗 | 输入、输出和合计使用记录 |
| 证据等级 | 证据不足、基本充分、充分 |
| 变化说明 | 优化前后差异解释 |

表格记录的是研究过程证据，不代表设置固定通过条件。

## 固定通过条件说明

检索准确率、响应延迟和 Token 消耗用于支持“测试、优化、再测试、对比展示”的研究流程。文档不规定固定通过条件，也不以某个具体数值决定项目是否成立。

## 证据等级

系统评估只使用以下三种证据等级：

| 等级 | 含义 | 判定语义 |
|---|---|---|
| 证据不足 | 证据链缺口明显，无法支持关键结论 | 只能说明做过验证动作，不能说明结论成立 |
| 基本充分 | 覆盖核心路径并能支撑主要结论，但边缘或异常覆盖有限 | 可支持目标基本成立，保留已知不确定性 |
| 充分 | 关键路径、关键约束与主要异常面均有一致证据支撑 | 可支持结论稳健成立，剩余风险可陈述且可管理 |

## 禁止的评估写法

- 不写固定性能门槛。
- 不把某个具体数值作为通过条件。
- 不记录每次运行流水。
- 不把研究性指标替代功能验收。
