# 项目开发规格

## 项目定位

本项目是面向单个个人用户的桌面端本地知识库助手。系统的核心价值是帮助用户围绕本地资料进行问答，并在回答中展示可追溯来源。

系统不是通用聊天机器人，不是网页系统，不是多用户协作平台，也不是通用智能体平台。MCP 在本项目中用于组织能力接入和服务边界，服务于文档解析、OCR、检索、LLM 调用等能力的解耦。

## 目标用户与使用场景

目标用户是需要整理和查询本地学习、科研资料的个人用户。

核心使用场景：

- 用户导入本地 PDF、Markdown、TXT 资料。
- 用户通过聊天式交互询问资料内容。
- 系统从本地知识库召回相关片段。
- 系统基于召回片段生成回答。
- 用户查看回答依据、文档名、位置和引用片段。

## 系统闭环

系统必须形成以下闭环：

本地文档导入 → 统一解析 → 文本清洗 → 结构感知切块 → 索引构建 → 用户提问 → 检索相关 Chunk → 组装上下文 → 调用大模型 → 生成回答 → 返回回答与引用。

实现不得跳过 IR、检索或引用返回环节。

## 功能性需求

### 文档导入

- 系统必须支持 PDF、Markdown、TXT。
- 对其他格式必须拒绝导入并给出明确错误提示。
- 系统必须为导入文档建立可追踪元数据和透明处理状态；字段与状态枚举以 `data_contracts.md` 为唯一规范源。

### 文档解析与 IR

- 所有支持格式必须转换为统一 IR。
- 后续清洗、切块、索引、检索和引用都必须依赖 IR。
- 各格式解析后的块字段、结构语义和位置字段以 `data_contracts.md` 为唯一规范源。

### 文本清洗

- 系统必须执行规则化清洗，去除无效文本并保留可用于切块和引用的结构信息。
- 清洗不得破坏标题路径、原文顺序、页码或等价位置等回溯依据。
- 清洗规则涉及的字段、不变量和可回溯要求以 `data_contracts.md` 为唯一规范源。

### 结构感知切块

- 切块输入必须是 IR 块序列。
- 系统必须先依据结构边界组织内容，再依据长度约束合并或拆分。
- 每个 Chunk 必须能回溯到原始块集合，并保留来源文档与位置映射。
- Chunk 字段、长度约束和回溯不变量以 `data_contracts.md` 为唯一规范源。

### 索引与检索

- 系统必须建立面向 Chunk 的本地检索索引。
- 用户问题必须转换为可检索查询表示。
- 检索必须返回相关 Chunk 集合。
- 检索对象是 Chunk，不是整篇文档。
- 召回数量等检索参数以 `data_contracts.md` 和 `module_interfaces.md` 的对应契约为准。

### 检索增强问答

问答流程必须包含：接收问题、构造检索查询、召回相关 Chunk、组装上下文、调用大模型、生成回答、返回回答与引用。

禁止未检索直接回答本地知识库问题。禁止只返回模型结论而不附带引用。禁止默认把整篇文档全文塞入提示词。证据不足时不得伪造确定性结论。

### 引用展示

- 每次回答必须返回回答正文和引用列表。
- 引用对象字段和回溯链路以 `data_contracts.md` 为唯一规范源。
- 用户界面中必须能看到回答正文、引用来源列表、每条引用对应的文档名与位置。

### 会话管理

系统必须支持多个会话。会话用于管理问答历史，不用于用户隔离。每个会话至少保存会话标识、创建时间、消息历史、关联回答记录。

### OCR 规则

当 PDF 页面无法提取有效文本时，允许调用现有 OCR 工具补充文本。OCR 输出必须进入统一 IR，不得形成独立于主流程的平行数据结构。系统不自行编写 OCR 算法。

## 非功能性需求

- 系统必须采用分层结构，至少包含表现层、应用层、领域层、基础设施层。
- 业务流程不得堆积在 UI 层或单一脚本中。
- 文档存储、解析、清洗、切块、索引建立和检索必须在本地完成。
- 外部大模型 API 只允许接收用户问题、检索得到的相关 Chunk、必要系统提示词。
- 禁止默认发送全量文档、全部知识库内容或与问题无关的原始材料。
- LLM 调用必须通过独立服务接口封装。
- API Key 必须通过环境变量读取。
- 普通单次提问必须完成检索、回答、引用返回的完整链路。

## 非目标与禁止范围

以下内容不属于本项目实现范围：

- DOCX 及其他未列明文档格式。
- 摘要生成功能。
- 多用户账号系统。
- 云端同步。
- 企业级权限管理。
- 团队协作知识库。
- 多智能体协同框架。
- 插件商店。
- 工作流引擎。
- 通用编程代理平台。
- 面向互联网公开部署的 SaaS 系统。
- 将 MCP 扩展为完整通用协议生态。
- 绕过 IR 处理后续流程。
- 绕过 RAG 检索链路直接回答本地知识库问题。
- 取消来源引用展示。

## 技术硬约束

- 使用 Python 实现完整业务逻辑。
- LLM 接入必须使用 OpenAI-compatible API。
- API Key 从环境变量读取。
- OCR 只集成现有工具。
- 文档格式仅 PDF、Markdown、TXT。
- 业务层依赖抽象接口，不直接绑定具体厂商 SDK 调用细节。
- 本地处理边界必须清晰。

## 验收标准摘要

系统成功原则如下：

- 能导入 PDF、Markdown、TXT。
- 能将导入文档转换为统一 IR。
- 能基于 IR 生成结构感知 Chunk。
- 能对用户问题召回相关 Chunk。
- 能基于召回结果生成回答。
- 能返回引用；证据不足时能明确说明依据不足。
- 桌面端界面具有会话区、主问答区、引用区。
- 模块边界清晰，输入输出对象明确。
- 任一引用能追溯到对应文档和位置。
