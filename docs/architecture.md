# 架构设计

## 架构目标

系统架构服务于单用户桌面端本地知识库助手。架构必须保证文档处理、检索、问答、引用展示之间的边界清晰，便于后续替换解析器、索引库、Embedding 服务、LLM 服务或 OCR 工具。

架构设计必须避免把业务规则堆积到 UI 层，也不得把系统写成单文件原型后视为最终结构。

## 分层结构

系统至少包含四层：

| 层级 | 职责 | 不承担职责 |
|---|---|---|
| 表现层 | 桌面端界面、用户输入、回答展示、引用展示、会话切换 | 文档解析、检索算法、LLM 调用细节 |
| 应用层 | 编排导入、解析、清洗、切块、索引、问答、引用映射流程 | 具体文件解析实现、索引库内部实现 |
| 领域层 | Document、DocumentBlock、Chunk、Citation、AnswerResult、Session 等核心对象与业务规则 | UI 控件、厂商 SDK 调用 |
| 基础设施层 | 文件访问、解析器、OCR 工具接入、索引存储、Embedding、LLM API 封装 | 业务流程决策 |

依赖方向应从表现层指向应用层，应用层指向领域层和抽象基础设施接口，领域层不依赖 UI 或具体外部服务。

## 核心数据流

1. 用户通过桌面端选择本地文档。
2. 应用层创建文档记录并触发解析。
3. 基础设施解析器将 PDF、Markdown、TXT 转换为统一 IR。
4. 应用层调用清洗规则处理 IR。
5. 切块逻辑基于 IR 生成 Chunk。
6. 检索基础设施为 Chunk 建立索引。
7. 用户在会话中提出问题。
8. 应用层构造查询并召回 Top-K Chunk。
9. 应用层将问题、召回 Chunk 和必要系统提示词传给 LLM 服务。
10. LLM 服务通过 OpenAI-compatible API 获取回答。
11. 引用映射逻辑将回答依据映射为 Citation。
12. 表现层展示回答正文和引用列表。

## MCP 角色定义

MCP 在本项目中是能力接入与组织层。它用于为解析、OCR、检索、LLM 调用等能力定义清晰输入输出，降低业务流程对具体工具实现的耦合，并为替换底层实现提供边界。

MCP 不代表通用 Agent 平台，不承担多智能体协作，不承担任意插件热插拔，不承担工作流编排平台职责。

## 本地处理边界

以下能力必须在本地完成：

- 文档存储。
- 文档解析。
- 文本清洗。
- 结构感知切块。
- 索引建立。
- 检索。

本地知识库内容不得默认整体发送到外部服务。

## 外部 API 边界

外部大模型 API 只允许接收：

- 用户问题。
- 检索得到的相关 Chunk。
- 必要系统提示词。

禁止默认发送全量文档、全部知识库内容、与当前问题无关的原始材料。LLM 服务必须通过独立接口封装，业务层不得直接依赖具体厂商 SDK 调用细节。

## UI 与业务逻辑边界

桌面端 UI 负责显示和交互：

- 会话列表与会话切换。
- 用户问题输入。
- 用户消息与系统回答展示。
- 引用来源展示。
- 文档导入入口与处理状态提示。

UI 不负责：

- 解析文档。
- 清洗文本。
- 生成 Chunk。
- 建立索引。
- 执行检索。
- 调用 LLM API。
- 生成引用映射。

## 禁止的架构形态

- 单文件堆叠全部逻辑。
- UI 层直接调用解析器、索引库或 LLM API。
- 业务层直接绑定具体厂商 SDK。
- 绕过 IR 的清洗、切块、索引或引用流程。
- 绕过检索直接生成本地知识库回答。
- 将 MCP 描述为通用 Agent 平台或多主体协作平台。
- 将桌面端应用改写为纯命令行工具、纯后端服务或纯网页原型。
